# M6.5 (v-stdlib slice) — VSLSEC SecID→#200 binding

The `v` half of the M6.5 auth stack — see the architecture in m-stdlib
[[m6.5-stdjwt]] (validate the TOKEN, not the PIV card) and the original VSLSEC in
[[m4-vslsec-vsllog]]. Token validation is engine-neutral (`STDJWT`, `m`); this is
the VistA-specific identity binding (`v`).

## What shipped (VSLSEC)

- **`$$bySecid(secid)`** — the SecID → NEW PERSON (#200) IEN binding via
  **`EN1^XUPSQRY`** (RPC `XUPS PERSONQUERY`, **ICR 4575, Controlled
  Subscription**, custodian XU). This is the documented way a federated subject
  (the SecID claim of a validated SSOi/STS token) maps to a local #200 IEN —
  exactly what Kernel's own `XUSAML`→`$$FINDUSER^XUESSO2` do internally. Returns
  the IEN, or `""` on no-match / on a bare engine. Empty SecID → loud
  `,U-VSL-SEC-ARG,` (the VSLSEC malformed-call contract).
- **`$$parseQry(result)`** (internal, pure) — extracts the IEN from the
  XUPSQRY by-ref result array. Bare-engine unit-testable; this is the logic
  half that proves dual-engine without a live VistA.

## EN1^XUPSQRY calling convention (corpus-grounded — non-obvious)

Spec from the vdocs GOLD corpus (`XU/krn_8_0_dg_common_services_ug/en1xupsqry-query-new-person-file`):
`EN1^XUPSQRY(result,xupsecid,xupslnam,...)` — **9 params**. Pass the SecID in
**param 2** and a **null last name in param 3** to query by SecID (no mode flag —
the populated param selects the mode). It does **NOT** return a string — it
**stuffs a by-ref array** (materializes at `^TMP($J,"XUPSQRY",...)`, read back via
the by-ref local):
- `result(1)` = found flag, **1 found / 0 not found** (the no-match signal).
- `result(1,0)` = `VPID^IEN^LastName~First~Middle^SSN^DOB^SEX^` — **the #200 IEN
  is ^-piece 2** (name sub-fields joined by `~`, not `^`).

No direct `^VA(200` read → waterline holds (the IEN comes from the
Controlled-Subscription API's returned array). There is **no Supported
(non-CS) SecID→IEN API**; `$$IEN^XUPS` (Supported) only does VPID→IEN, and the
`ASECID` xref / `^VA(200,IEN,205)` piece layout are **NOT in the gold corpus**
(don't hard-code them). SecID is #200 field **#205.1** (corpus-confirmed).

## Gotcha — VSLSECTST is now dual-engine

`setup` previously called `$$DT^XLFDT` (Kernel, absent on the bare
m-test-engine) → the suite aborted **0/0** on a bare engine (it was live-only).
Fixed: `set DT=$select($text(DT^XLFDT)'="":$$DT^XLFDT,1:3000101)`, and
`tDuzAndUser`'s `$$user` (FileMan) soft-skips when `$text(GET1^DIQ)=""`. So the
pure-logic + arg-contract tests now run on both bare engines (M6.4 pattern); the
VistA bindings (`$$user`, `$$bySecid`) run live. The live `$$bySecid` no-match is
**provable without mutation** (a bogus SecID → ""); a positive match soft-skips
(SecIDs are IAM-provisioned, not seeded on the test builds).
